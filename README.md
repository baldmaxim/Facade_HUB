# Автозаполнение ВОР по фасадным работам

Автоматическая расценка Ведомости Объёмов Работ (ВОР) для фасадных конструкций: НВФ, СПК, мокрый фасад, утепление, ограждения, двери, козырьки. SPA на React 19 + Vite 7, бэкенд — Supabase, AI-ревьюер — Gemini 2.5 Flash через Edge Function.

---

## Как это работает

1. Пользователь загружает пустой ВОР заказчика (Excel) на странице **Заполнение ВОРа**.
2. Парсер извлекает позиции, разделы и примечания.
3. Матчер подбирает фасадные шаблоны по ключевым словам (включая примечание заказчика).
4. Генератор строит заполненный Excel: работы, материалы, коэффициенты `j`/`k`, пути затрат, цвета строк.
5. (Опционально) AI-ревью через Gemini оценивает каждую распознанную позицию: вердикт + score + комментарий.
6. Файл скачивается + автоматически попадает в историю объекта.

```
src/lib/vorMatcher.js               → matchPositionDetailed, isHeader, classifyRowRole, detectVorStyle
src/lib/rules/vorRules.js           → MATCH_RULES (приоритетный массив правил)
src/lib/vorTemplates.js             → 36+ код-шаблонов (источник истины)
src/lib/vorExcelGenerator.js        → парсинг и генерация Excel (simple + split-3)
src/lib/vorPriceLoader.js           → парсер прайса работ + 4-ступенчатый лукап
src/components/VorFillModal.jsx     → модалка заполнения (5-этапный пайплайн + AI)
src/components/VorPipelineSteps.jsx → индикатор этапов
src/components/VorReviewRow.jsx     → раскрывашка AI-разбора + 👍/👎
src/components/VorAiPanel.jsx       → панель управления Gemini-ревью/предложений
src/pages/VorPage.jsx               → страница объекта: история ВОР + кнопка модалки
src/pages/VorTemplatesPage.jsx      → база шаблонов (read-only код + редактируемые custom)
supabase/functions/vor-review/      → Edge Function: Gemini 2.5 Flash + RAG
```

## Шаблоны фасадных работ

**СПК (светопрозрачные конструкции):**
- `spk_profile` — профиль стойка-ригель
- `spk_glass` — заполнение стеклопакетом
- `spk_broneplenka` — бронирующая плёнка
- `spk_hardware` — фурнитура
- `pvh_profile` — ПВХ-окна
- `doors_entrance` — входные двери (БКФН, витражные)
- `doors_tambour` — тамбурные двери (отдельный costPath)
- `custom_stvorka_spk` — авто-подмешивается к окнам/витражам со створками (custom-шаблон в Supabase)

**НВФ (навесной вентилируемый фасад):**
- `nvf_subsystem` — подсистема НВФ
- `insulation` — утепление (универсальная толщина, 2 слоя, мембрана всегда)
- `nvf_cladding_clinker` — клинкерная плитка (2 работы: облицовка + затирка FM.R-E), k=1.22
- `nvf_cladding_cassette` — алюминиевые кассеты, k=1.22
- `nvf_cladding_fibrobeton` — стеклофибробетон, k=1.22
- `nvf_cladding_ceramic` — керамические панели, k=1.22
- `nvf_cladding_porcelain` — керамогранит, k=1.22
- `nvf_cladding_natural_stone` — натуральный камень, k=1.22
- `nvf_cladding_akp` — алюмокомпозитная панель, k=1.22
- `nvf_cladding_concrete_tile` — бетонная плитка, k=1.15
- `nvf_cladding_brick` — кирпич облицовочный, k=1.05
- `nvf_cladding_fcp` — фиброцементная панель, k=1.05
- `nvf_cladding_galvanized` — оцинкованный лист, k=1.20
- `nvf_cladding_arch_concrete` — архитектурный бетон, k=1.00
- `nvf_cladding_profiles_vertical` — вертикальные прямоугольные профили (2 работы)

**Мокрый фасад:**
- `wet_facade` — полный мокрый фасад ТЕХНОНИКОЛЬ (ROCKforce + ROCKglue + утеплитель inline + дюбель + ROCKmortar + сетка + ROCKprimer + штукатурка + ROCKsil)
- `wet_facade_insulation` — только клеевое утепление (fallback)
- `wet_facade_finish` — декоративный слой / штукатурка (fallback)
- `wet_facade_paint` — окраска

**Ограждения и козырьки:**
- `glass_railing` — стеклянные ограждения (2 EPDM уплотнителя + анкерный крепёж Ø14х110)
- `glass_railing_molled` — моллированные ограждения (+ гибка профиля)
- `glass_canopy` — козырьки из триплекса

**Прочее:**
- `flashings` — отливы / парапеты (2 работы: оцинковка + герметик СТИЗ-А/В)
- `pp_otsechi` — П/П отсечки (для откосов; шаблон идентичен flashings, разные `noteGp`)
- `vent_grilles` — фасадные вентиляционные решётки
- `scaffolding` — леса и люльки
- `kmd_spk`, `kmd_nvf` — разработка КМД (800 / 400 руб/м²)
- `mockup` — мокап фасада

## Умная логика

**Автодетекция толщины утеплителя НВФ** по названию/примечанию (`detectInsulationThickness`):
- Паттерны: `NNN мм`, `толщ.NNN`, `утеплителем NNN`, габариты `x NNN`.
- **Требует слово** «утепл / теплоизол / изоляц / минват / каменн.ват / базальт» в ~30 символах до числа — иначе число игнорируется (исправлено в Фазе 5: ранее `0.03 м³` → 70 мм; теперь корректно идёт fallback на 180 мм).
- Формула: наружный слой всегда **50 мм** (`j=0.05`), внутренний = `(толщина−50)/1000`. При T ≤ 50 мм — один слой.
- Поддержка формата `"X+Y мм"` через `detectInsulationLayers()`.
- Противопожарная мембрана ТехноНИКОЛЬ АЛЬФА ПРО НГ — **всегда присутствует**.

**Поле `defaultThickness` в правилах** — резервная толщина, используется генератором как fallback перед хардкодом 150 мм. Применено в правиле решёток (180 мм).

**Утеплитель в мокром фасаде** — вставляется inline в генераторе между ROCKglue и дюбелем (не как отдельный secondary). Толщина и тип — из названия позиции.

**Вторичные шаблоны** (scaffolding, КМД) добавляются автоматически к СПК/НВФ позициям. Если в ВОР есть отдельная позиция «Леса» или «Разработка КМД» — secondary из других позиций **исключается** через `excludeFromSecondary` (нет дублей).

**Авто-подмешивание шаблонов в `runRules`:**
- Откос («откос») к НВФ-облицовке → добавляется `pp_otsechi`.
- Отлив («отлив») к НВФ → добавляется `flashings`.
- Окно/витраж со створкой (`spk_profile|pvh_profile` + регэксп `створ|откидн|одностворч|двустворч|поворот`) → добавляется `custom_stvorka_spk`.

**Пропускаемые позиции:**
- Перголы — всегда ручная расценка.
- Пол в технических лоджиях — не наш раздел.
- Раздвижные двери, металлические ограждения парапета (Фаза 2).

## Два стиля ВОР — авто-детекция

`detectVorStyle()` смотрит на маркер «Прочие материалы»:

- **Simple** (Сокольники, Событие, ВГК5, Адмирал): работы и материалы идут напрямую за позицией.
- **Split-3** (Муза): позиции разделены на роли — **work** (работа), **material** (отдельный материал по unit), **auxiliary** («Прочие материалы», собирает мембрану + крепёж из шаблонов кластера).

В split-3 генератор:
- `clusterMaterialUnits` — Set unit-ов material-позиций кластера.
- В **material**-позиции `«Утеплитель толщ. X мм»` (unit=м³) идёт только утеплитель. Мембрана с unit=м² фильтруется.
- В **auxiliary**-позиции «Прочие материалы» собираются все материалы шаблонов кластера с unit, отсутствующим в `clusterMaterialUnits` (мембрана + 2 разных дюбеля).
- Дедуп по ключу `name|unit|j|k`.
- Сброс `clusterMaterialUnits` + `clusterTemplates` на auxiliary-границе.

## Приоритет правил матчинга

Специфичные — первыми, общие — последними. Полный массив — в `src/lib/rules/vorRules.js`.

1. Пропускаемые: освещение фасада, перголы, пол в тех. лоджиях, раздвижные двери, металлические ограждения парапета.
2. Мокап, фурнитура, КМД.
3. Защитные экраны, ограждения, моллированные.
4. **Откосы с мокрым фасадом** → `wet_facade + pp_otsechi`.
5. **Откосы с кассетами** → `nvf_subsystem + cassette + pp_otsechi`.
6. **Откосы из облицовки без утеплителя** (Фаза 5) → как навесные на витраж + отсечки, без insulation.
7. **Навесные на витраж / пилоны** → `nvf_subsystem + cassette` (без утепления).
8. **Короба кондиционеров** → `nvf_subsystem + cassette` (без утепления).
9. **Решётки под лоджиями** (`решётки`/`решетки`) → полный НВФ с кассетами. Стоит **выше** «корзин А/С», иначе «решётки над нишами кондиционеров» уходят в гибрид wet+НВФ.
10. **Корзины А/С** (Фаза 5: чёткое разделение от ниш) → гибрид `wet_facade + nvf_subsystem + nvf_cladding_cassette + scaffolding + kmd_nvf`. Мокрый — оштукатуривание внутренних откосов ниши, кассета — лицевая плоскость.
11. **Стемалит как облицовка НВФ** → гибрид `spk_profile + insulation + spk_glass + kmd_nvf` (без `nvf_subsystem`).
12. **Вертикальные прямоугольные профили** → `nvf_cladding_profiles_vertical`.
13. **Окно/окном** (regex `окно`, **выше** дверей) — узкое правило, чтобы «Окно одностворчатое» не перехватывалось дверьми через keyword `створч`. `окно` matches «окно/окном», но не «оконных/оконный».
14. Тамбура, двери, окна — **до** стоечно-ригельн.
15. **Козырёк** — **выше** стоечно-ригельн (включая стальные козырьки → `glass_canopy` через имеющееся правило).
16. НВФ по типу облицовки: клинкер, кассеты, фибробетон, керамика, камень, АКП, бетонная плитка, кирпич, ФЦП, оцинковка, арх. бетон.
17. **Полный мокрый фасад** (`wet_facade`) — **до** `wet_facade_insulation` и штукатур.
18. Штукатур / мокрый фасад fallback → `wet_facade_finish`.
19. Утеплитель, леса, подсистема, витраж — общие fallback.

**Custom-правила** в `vor_custom_templates` (Supabase) — fallback после кодовых правил. Первое совпадение побеждает.

## Структура Excel ВОР заказчика

### Столбцы (A–T)
| Стб | Имя | Назначение |
|---|---|---|
| A | Номер позиции | `10.1.1.1.` (Excel может распознать как дату — обрабатывается парсером) |
| B | № п/п | Порядковый номер |
| C | Затрата на строительство | Путь раздела (см. **Пути затрат** ниже) |
| D | (флаг учёта) | `да` для материалов |
| E | Тип элемента | `суб-раб` / `суб-мат` / пусто |
| F | Тип материала | `основн.` / `вспомогат.` / пусто |
| G | Наименование | Описание работы/материала |
| H | Ед. изм. | м², шт, м.п., м³ |
| I | Количество заказчика | Объём от заказчика |
| J | Коэфф. перевода | Для материалов |
| K | Коэфф. расхода | Для материалов |
| L | Количество ГП | Объём генподрядчика |
| M | Валюта | RUB |
| N | Тип доставки | `в цене` / `не в цене` |
| O | Стоимость доставки | (обычно пусто) |
| P | **Цена за единицу** | Главное поле расценки |
| Q | Итоговая сумма | `L * P` |
| R | Ссылка на КП | URL коммерческого предложения |
| S | Примечание заказчика | Учитывается матчером |
| T | Примечание ГП | Заполняется генератором (`noteGp`) |
| U | **Проверка АИ** | 21-я колонка, добавляется только если AI-ревью запускали |

### 5 типов строк
1. **ЗАГОЛОВОК** (серый `FFE0E0E0`) — имена столбцов (строка 1).
2. **РАЗДЕЛ** (без цвета) — `A`=код раздела, `G`=название, `Q=0`. Пример: «Фасады», «Окна и витражи».
3. **ПОЗИЦИЯ** (розовый `FFFFCCCC`) — `A`=код, `G`=описание, `H`=ед., `I`=кол-во заказчика, `L`=кол-во ГП. Цена `P` пуста.
4. **РАБОТА** (сиреневый `FFE6D9F2`) — `E="суб-раб"`, `G`=вид работы, `L`=объём, **`P`=цена**.
5. **МАТЕРИАЛ** (зелёный `FFE8F5E0`) — `E="суб-мат"`, `F`=тип, `G`=материал, `J`/`K`=коэффициенты, **`P`=цена**.

Формулы (значения, не Excel-формулы): `L_материала = L_позиции * J * K`, `Q_работы = L * P`.

## Портал — функции

### Страница объекта (`VorPage`)
- Кнопка **«Заполнение ВОРа»** → модалка `VorFillModal`.
- **История ВОР** — список ранее сгенерированных версий (download / delete).

### Модалка `VorFillModal` — 5-этапный индикатор пайплайна
После выбора файла автоматически прогоняются:
1. **Парсинг файла ВОР** — извлечение позиций и разделов.
2. **Загрузка контекста** — прайс из БД / нового файла, custom-шаблоны и custom-правила.
3. **Подбор шаблонов** — `matchPositionDetailed` для каждой позиции (с учётом override).
4. **Генерация Excel** — dry-run в память, blob кешируется в `cachedResult`.
5. **Готов к скачиванию** — кнопка «Расценить и скачать» работает мгновенно.

Любая смена опций (Донстрой, прайс, ручные override, AI-ревью) перезапускает этапы 2–5 автоматически. Сохранение в историю и в БД (если новый прайс) — молча на скачивании.

### Превью матчинга
- Чипы шаблонов с tooltip: имя сработавшего правила.
- Кнопка ✏ на каждой позиции — попап с галками всех шаблонов (override per-position).
- Чип `⚙ из кластера` — для auxiliary-позиций в split-3 (собирается генератором).
- Чип `⚡` — сработало custom-правило из Supabase.
- AI-вердикт: цветной кружок + score, клик раскрывает разбор.
- Кнопки 👍/👎 + форма правильной разметки — отправляет фидбек в `vor_review_feedback`.

### Страница «База шаблонов» (`VorTemplatesPage`)
- 36 код-шаблонов из `src/lib/vorTemplates.js` — read-only.
- Custom-шаблоны из `vor_custom_templates` (Supabase) — редактируются через `VorCustomTemplateEditor`.
- Custom-шаблоны fallback после кодовых.

### Донстрой-режим
Флаг `priceAllWithQty` в `isHeader`. Для Донстроя (Событие, Символы) расцениваются и родители, и дочерние позиции. Для остальных (Муза, Сокольники) композитный родитель = заголовок.

### Прайс работ (per-object)
Таблица `vor_work_prices` (миграция 020). 4-ступенчатый лукап в `findWorkPrice`:
1. Точное совпадение `tplKey + name`.
2. Единственная запись на `tplKey`.
3. Префикс по первому слову.
4. Fallback `costPath + name`.

## AI-ревью подбора шаблонов (Gemini 2.5 Flash)

**Архитектура:**
- Клиент: `src/components/VorFillModal.jsx` — кнопка «Проверить подбор AI» прогоняет распознанные позиции последовательно (rate-limit).
- Edge Function: `supabase/functions/vor-review/index.ts` — прокси к OpenRouter, ~1 МБ (бандлит `@supabase/supabase-js`).
- Модель: `google/gemini-2.5-flash`, temperature 0.2, max_tokens 700.
- Секрет функции: `OPENROUTER_API_KEY` (Dashboard → Edge Functions → Secrets).
- JWT verification **включён** (дефолт). Клиент использует **legacy anon JWT** (см. подвох с ключами ниже).

**Схема запрос/ответ:**
```
IN  POST { noteCustomer, tplKeys[], posCode?, costPath? }
OUT 200  { reasoning, score (0..100), verdict, comment, posCode, tokens }
```

Вердикт выводится из score: `≥70 → green`, `40..69 → yellow`, `<40 → red`. При ошибке `score: 50, verdict: 'yellow', comment: 'Отладка: <причина>'`.

**Ключевые приёмы:**
1. **Chain-of-Thought** — поле `reasoning` стоит ПЕРВЫМ в JSON, модель обязана рассуждать перед оценкой.
2. **In-context reference** — каталог 35 шаблонов + 11 правил + 14 эталонных примеров + 5 частых ошибок (из `sobytie_correct.txt`, `sokolniki_comparison.txt`, `muza_final_comparison.txt`).
3. **Score 0–100** вместо verdict+confidence — одно число = качество подбора.
4. **RAG** — перед каждым вызовом RPC `find_similar_feedback` (pg_trgm similarity по `note_customer + pos_name`) подмешивает top-3 прецедента в user-message как «ПОХОЖИЕ СЛУЧАИ ИЗ ИСТОРИИ».
5. **UI-фидбек 👍/👎** → запись в `vor_review_feedback` (форма с 35 шаблонами при 👎).
6. **Excel-колонка U** «Проверка АИ»: `🟢 Оценка: 95/100 — комментарий\n\nРазбор: <текст>`.

**Стоимость:** ~2500 input + 150–250 output токенов на позицию. ВОР из 100 позиций ≈ $0.10 (см. сравнение моделей ниже).

**Деплой:**
```bash

```

### Roadmap AI-ревьюера (по убыванию пользы)
1. **Калибровка SYSTEM_PROMPT** — после 20–50 разметок пользователем выявляются системные правила, которые надо в base-prompt, а не в DB.

### Подвох с ключами Supabase

**НЕ класть в .env:** `sb_publishable_*` (не работает с Edge Functions), `Legacy JWT Secret` (серверный ключ — утечка = взлом).

## Известные подвохи

### `xlsx-js-style` shim в браузере
`xlsx-js-style/dist/xlsx.min.js` содержит `require("stream").Readable` и ссылки на `fs`/`crypto`. Vite экстернализует Node built-ins → ошибка `Module stream has been externalized... Cannot access stream.Readable in client code`.

**Решение:** в `vite.config.js` есть `resolve.alias` для `stream`/`fs`/`crypto` → `src/shims/empty-node.js` (`export default {}`). Проверки `is.Readable && bu(is.Readable)` падают в false и ветка пропускается.

**Поддержка:** при появлении новой библиотеки с `require("fs")` — добавить алиас в тот же блок. После обновления зависимостей и возврата ошибки — очистить `node_modules/.vite`. Shim — ровно 1 строка, не раздувать.

### Имя колонки `ai_confidence` в `vor_review_feedback`
С 2026-04-22 хранит **score 0–100** (качество подбора), не confidence. Имя не переименовано чтобы избежать миграции. При большом накоплении — сделать `ALTER TABLE ... RENAME COLUMN ai_confidence TO ai_score`.

## Архитектурные решения

- Код-шаблоны в `src/lib/vorTemplates.js` — **источник истины** (36 шт., не редактируется на портале).
- Custom-шаблоны в Supabase `vor_custom_templates` — fallback, редактируются через `VorCustomTemplateEditor`.
- Матчинг: `runRules` сначала по `MATCH_RULES`, потом по custom-правилам — первое совпадение побеждает.
- Генератор принимает `{ customTemplates, customRules, overrides, reviews, workPrices, priceAllWithQty }`.
- AI-ревью на сервере, не client-side: ключ OpenRouter не должен светиться в браузере.
- Леса + КМД оставляем в **конце** вывода (не переупорядочиваем).
- Code splitting (bundle 2040 → 601 KB), `ErrorBoundary` с fallback UI, SWR для кэша на 3 страницах.

## Ограничения

- Цены **основных** материалов не заполняются — прайсинг из КП идёт отдельно.
- Бренды в шаблонах — плейсхолдеры (`по проекту`), кроме фиксированных (затирка FM.R-E, дюбели с ценами).
- Двери в составе витража: в одних тендерах профиль учитывается в строке, в других — уже включён. Правится вручную.
- Имя материала «Муза ТЕХНОВЕНТ СТАНДАРТ» / «EJOT H5 155/255» вместо наших дефолтов — ручная подстановка заказчика.
- В split-3 толщина 150→180 мм у Музы (везде 180 несмотря на исходное описание) — ручной override на портале.
- В код-шаблонах `j=1/k=1.1` для створок СПК — плейсхолдеры. Точные нормативы — отдельной задачей.
- Отдельный шаблон `nvf_cladding_stemalit` НЕ создаётся — стемалит реализован как гибрид.
